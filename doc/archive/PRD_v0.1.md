# vibeCoding 工作流产品需求文档 (PRD v0.1)

## 1. 产品概述
### 1.1 背景
当前AI辅助编程已经成为趋势，但由于缺乏系统化的流程和高视角的架构约束，经常变成碎片化的“问答式代码生成”。针对这一痛点，我们希望打造一款“vibeCoding”工作流驱动产品。

### 1.2 目标与定位
本产品旨在建立一套标准化的自动化/半自动化软件开发生命周期。通过提供高质量、结构化的**AI友好型需求文档**，驱动主流大语言模型（如 Claude, Codex, Gemini 等）无缝衔接完成“需求理解 -> 代码编写 -> 测试验证 -> 版本代码提交”的全链路闭环流程。

## 2. 核心用户故事
- **需求定义阶段**：作为产品规划人员，我可以通过与AI对话，让AI帮我提炼、整理并输出一份逻辑严密、可被其他AI直接理解和执行的需求说明文档。
- **自动化编码阶段**：作为主开发人员，我希望AI能够无缝读取上述需求文档，并按照既定架构与模块设计，自主搭建项目代码。
- **测试与质量保证阶段**：作为测试验证人员，我希望在AI编码完成后，系统能自动运行测试流程（包括编译、审查和自动化测试），如果发现Bug能触发机器自启复（Self-healing），直至功能达到可用标准。
- **交付与存档阶段**：作为项目管理者，我希望在一个功能点（Feature）或Bug修复完全闭环后，AI能够自动化将代码封装，并提交保存至Git代码库中。

## 3. 标准工作流定义 (The vibeCoding Flow)

### Phase 1: 需求定义 (Idea -> Spec)
- **输入**：用户的一句话需求、草图或零散的想法。
- **处理**：AI作为产品经理，通过主动询问澄清边界条件、异常流和关键业务逻辑。
- **输出**：存放于 `/doc` 目录下的系统化需求文档（Markdown格式），包含业务逻辑、API设计草案及测试验收标准。

### Phase 2: 编码实现 (Spec -> Code)
- **输入**：Phase 1 产出的需求文档。
- **处理**：AI作为全栈工程师，按照文档拆解任务，分步骤创建/修改代码文件。
- **输出**：可运行的源代码。

### Phase 3: 测试验证 (Code -> Validated)
- **输入**：已完成的源代码与验收标准。
- **处理**：自动执行单元测试或端到端验证，若遇到报错信息，AI自行阅读错误日志、修复代码并重新运行，直至测试完全通过。
- **输出**：无Bug、达到交付标准的功能模块。

### Phase 4: 交付存档 (Validated -> Repo)
- **输入**：测试完毕的可用代码。
- **处理**：AI根据Diff变化生成符合规范的 Git Commit Message，并执行 `git commit`。
- **输出**：稳定、可追溯的代码版本沉淀。

## 4. 后续探讨事项
为了将本流程彻底打通且落地，我们需要在后续版本中明确以下细节：
1. **统一文档结构**：定义一份适用于驱动AI的最佳需求文档模板。
2. **人工卡点机制 (Human-in-the-loop)**：如何在各阶段（比如需求转开发前、正式Commit前）加入人类确认环节，防止AI“跑偏”。
3. **测试兜底策略**：若AI在自我修复测试Bug的阶段陷入死循环，应制定怎样的重试阈值与失败降级策略。
