# 完整工作流 (Complete Workflow) 模块设计文档 v0.7

## 1. CCG 原生工作流的真实定义

经过对底层引擎源文件 (`ccg-workflow/templates/commands/workflow.md`) 的深入核对，**`/ccg:workflow` 确实是一套极为严谨且深度的 6 阶段协作流程**。
它并非我之前设想的简化版敏捷操作，而是一场由 Claude 充当“项目经理（编排者）”、Codex 充当“后端工程专家”、Gemini 充当“前端设计专家”的重型多模型协作战役。

**这 6 个真实执行阶段为：**

| 阶段序号 | 阶段名称 | 内部运转逻辑 (多模型如何互动) |
|---|---|---|
| **Step 1** | **🔍 研究与分析 (Research)** | **Claude 单独工作**：截获用户的粗糙 Prompt，执行深度理解和补充增强。调用检索工具搜集现有项目上下文，最后给需求打分（低于 7 分直接拦停退回）。 |
| **Step 2** | **💡 方案构思 (Ideation)** | **Codex + Gemini 并行发力**：Codex 输出后端技术可行性和架构风险；Gemini 并行输出 UI 交互和视觉可行性方案。 |
| **Step 3** | **📋 详细规划 (Planning)** | **三模协同共建**：Codex 制定后端接口逻辑规范，Gemini 设计前端组件架构，Claude 负责把两边缝合，最终输出落地执行计划表。 |
| **Step 4** | **⚡ 实施 (Execution)** | **纯实干阶段**：严格按照上一步定死的框架与项目现有规范进行代码编写（可能是 Claude 主控或分发）。 |
| **Step 5** | **🚀 代码优化 (Optimization)** | **二次并行视察**：代码写完后，Codex 负责审查代码的安全性和错误处理漏洞；Gemini 审查样式的还原度和可访问性。 |
| **Step 6** | **✅ 质量审查 (Review)** | **Claude 终审**：对照计划逐项打钩划去，跑通验证用例，生成最终汇总报告，抛出结果给用户。 |

## 2. 核心架构冲突与 UI 应对策略

虽然流程极其严密（甚至可以说是市面上最强大的 AI 工作流之一），但这也引出了 **CCG 官方 README 中明确提出的防坑警告**：

> **“大型需求根本不适合挂空挡一键跑到底”**

因为在这漫长的 6 个阶段中，每一次方案沟通、每一次代码审查，都在疯狂吃 Token 和上下文。如果是一个全栈的大型模块（比如“给我写个完整的电商后台”），跑到第 4 步时，早期的上下文可能就被截断遗忘了，最后导致跑崩、乱改文件。

因此，我们在桌面端 UI 的设计上，必须做到**既尊重这 6 步流程的震撼感，又要锁死用户的危险操作**：

### UI 呈现方案改版

1. **保留 6 步进度条，正名文案**：
   将 `WorkflowPanel.vue` 的 NSteps 组件更新为权威的官方术语：
   - `研究分析` ➜ `方案构思` ➜ `详细规划` ➜ `编码实施` ➜ `代码优化` ➜ `质量审查`
   *(这能让用户清晰感知到底发生了什么，消除漫长等待带来的不可控感)*

2. **强化拦截向导 (Wizard Modal)**：
   既然它是重型武器，我们就不能允许像聊天框一样随便回车就发车。
   继续保留并强化刚才做好的**弹窗初始化设计**，甚至可以增加一道确认墙：
   - 弹窗最醒目位置写明：“建议这套流程只用于独立的组件 / 单页面 / 闭环的小体系开发”。
   - 提供一个大号文本框收取“种子 Prompt”。

3. **执行控制台 (Live Logs) 改造**：
   原版命令行里是通过文本 `[模式：研究]` 这样的 Tag 来标识的。
   我们在桌面前端可以做一个流式日志监听器，当日志打印出特定的 `[模式：X]` 时，界面上的 6 步轨道灯自动亮起对应的卡点，做到极其丝滑的运行动画。
