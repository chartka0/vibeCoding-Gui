# v0.6 UI 架构升级：多项目大盘与状态隔离

## 核心改动
为了支持“多个项目并行且保留历史记录”的需求，从前端交互与架构层面进行了重构：

### 1. 结构变更：引入总览页
放弃了直接进入工作流工作态的单项目逻辑，改为：
- **`Overview.vue` (总览大盘)**：应用启动后的第一落脚点，以卡片形式展示所有存在的项目。由于底层桌面环境尚未完全打通，目前前端通过 Mock 支持卡片的生成。
  - 支持 `新建项目`
  - 支持调用 Tauri 原生对话框 `打开本地项目`
- **`WorkspaceView.vue` (工作区页)**：当用户在大盘选中某个具体项目后，才会装载该项目的独立面板组（工作流面板、诊断面板等）。
- **`App.vue` 路由接管**：通过 `activePage` 在宏观上控制展示 `Overview` 还是 `WorkspaceView`。

### 2. 状态隔离：引入 Pinia
为了在多面板中共享和隔离同一个项目的数据，引入了 Vue 的原生状态管理 `Pinia`：
- **`store/workspace.ts`**：
  - `workspaces`: 当前系统内已存在的项目列表
  - `currentWorkspaceId`: 用于记录用户当前进入的是哪个项目的标识
  - 内部预留了针对后端 Rust 调用的接口锚点（获取 Workspace 列表的通讯占位）

### 3. 组件级调整
- **`Sidebar.vue`**：移除旧版切换逻辑，增加全屏工作台模式下的**“返回工作区总览”**按钮。
- **`WorkflowPanel.vue`**：新增当前选中项目名称标识展示；在未选择项目时进行保护，拦截运行。
- **`HistoryPanel.vue` (NEW)**：全新面板，用于读取和展示当前隔离在当前项目 ID 下的执行阶段和输出记录表（暂用 Mock 数据占位预演）。

## 下一步 (Rust 后端支撑)
UI 前提条件组装完毕，下一阶段 v0.6.1 将实现底层并行和历史沉淀：
1. Tauri + `rusqlite` 构建本地存储，替代上述 Mock 的空间数据与历史 `workflow_runs`。
2. 利用 Rust `tokio` 协程池隔离运行各项目的 `ccg-workflow`，实现真正的并发跑批。
