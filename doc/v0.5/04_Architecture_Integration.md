# vibeCoding 工作流产品需求文档 (v0.5) - 核心引擎与可视化架构 (Tauri + CCG)

## 1. 战略重构理念：UI as a Shell, CCG as the Core
传统的 AI 辅助工具通常是“壳包核”，但我们产品的最新定位是：**将业界公认优秀的 CCG (Context-Creation-Generation) Workflow 作为我们的“底座内核”，而我们要打造的 Windows/Mac 桌面端，本质上是一个强大的“CCG 可视化调度与交互面板”。**

这一理念意味着：
- 我们**不重复造轮子**去发明如何让 AI 理解上下文。
- 我们的主要工程量在于：如何把 CCG 这套原本偏命令行的流程，优雅地映射到 **Naive UI** 的图形界面上，降低普通用户（非资深极客）的使用门槛。

## 2. 架构映射 (CCG -> Tauri Desktop)

### 2.1 Context 构建阶段 (Phase 1)
- **CCG 原理**：AI 收集上下文，进行澄清追问。
- **桌面端可视化方案**：
  - **组件应用**：引入类似 IDE 的“Chat 侧边栏”或“对话气泡”。
  - **交互设计**：当底层的 Claude 发起澄清追问时，Tauri 会在前端弹出一个**动态问卷(Dynamic Form)** 或带有确认按钮的浮层，而不是单纯的聊天。
  - **核心价值**：将长篇大论的 AI 追问，可视化为用户只需“点选”或“输入短文本”即可完成的表单。

### 2.2 Creation 阶段 (工程文档生成)
- **CCG 原理**：AI 生成 `UI_Spec.md`, `Logic_Spec.md`, `Test_Spec.md`。
- **桌面端可视化方案**：
  - **组件应用**：使用代码 / Markdown 高亮面板。
  - **交互设计**：左侧展示用户原始需求池，右侧使用类似 VSCode 的 Tab 标签页，并列展示生成的 3 份 Spec 文档。
  - **核心卡点**：底部放置一个醒目的 `⭐ 批准并开始编码 (Approve & Build)` 主按钮。不点击按钮，底层的 Codex/Gemini 进程将被挂起。

### 2.3 Generation 阶段 (Phase 2 编码执行)
- **CCG 原理**：根据规格说明书，由各个模型去写代码。
- **桌面端可视化方案**：
  - **组件应用**：全局进度条、控制台日志输出框 (Terminal View)、骨架树 (File Tree)。
  - **交互设计**：用户点击 Approve 后，界面切换至“构建监控模式”。
    - 控制台实时打印 `[Codex] 正在生成 xxx.js...`，`[Gemini] 正在组装 xxx.vue...`
    - 文件树组件实时高亮显示当前正在被 AI 修改或新建的文件。

## 3. 为什么要做可视化？(产品竞争壁垒)
完全基于 CLI 的 CCG 门槛极高，对非程序员的产品经理很不友好。我们的 Tauri 桌面端能够提供：
1. **进度透明化**：缓解等待 AI 写代码时的焦虑。
2. **审查具象化**：不需要去翻 `.md` 文件，所有任务说明都在精美的面板里并排对比。
3. **沉浸式控制**：随时中止、回滚、人工介入（Human-in-the-loop）就像打游戏一样直观。
